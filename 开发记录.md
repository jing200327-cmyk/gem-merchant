| 选项 | 决定 | 原因 |

## 游戏规则实现

### 核心循环
1. 设置阶段：选择人数 (2-4) + 输入玩家名
2. 主循环：轮流执行动作 → 检查贵族 → 检查终局条件
3. 切换机制：每回合结束弹出遮罩层，保护上一位玩家的隐私信息

### 四种动作
| 动作 | 实现 | 校验 |
|------|------|------|
| 拿 3 个不同色宝石 | `actionMode = 'take3'` | 宝石池有余量；玩家总数 ≤ 10 |
| 拿 2 个同色宝石 | `actionMode = 'take2'` | 目标堆 ≥ 4；玩家总数 ≤ 10 |
| 购买卡牌 | `actionMode = 'buy'` | 费用计算含折扣；黄金顶替缺口 |
| 预留卡牌 | `actionMode = 'reserve'` | 预留 ≤ 3 张；自动获得 1 黄金 |

### 关键算法

**购买费用计算** (`effectiveCost`):
```
实际费用 = max(0, 卡面费用 - 玩家已有同色卡牌数量)
```

**支付逻辑** (`executePayment`):
1. 先用匹配颜色的宝石支付
2. 剩余缺口用黄金 (万用宝石) 补齐

**贵族检查** (`checkNobles`):
- 每次购买后遍历贵族列表
- 检查玩家各色卡牌数量是否满足条件
- 满足则自动拜访 (+3 声望)

**终局判定**:
- 玩家达到 15 声望 → 标记 `finalRoundPlayer`
- 其余玩家各获最后一回合
- 轮回到 `finalRoundPlayer` 时游戏结束
- 平局判定: 声望相同 → 比卡牌数量 (少者胜)

### 宝石池配比
| 玩家人数 | 每色宝石 | 黄金 |
|----------|----------|------|
| 2 人 | 4 | 5 |
| 3 人 | 5 | 5 |
| 4 人 | 7 | 5 |

### 卡牌生成
- **一级** (40张): 费用 1-4, 声望 0-1
- **二级** (30张): 费用 5-7, 声望 1-3
- **三级** (20张): 费用 7-12, 声望 3-5
- 费用随机分布在 1-3 个颜色上，每色至少 1 点

### 贵族 (10 种预设，抽取 n+1)
- 双色类: 4+4 (5种)
- 三色类: 3+3+3 (5种)
- 每个价值 3 声望

```

    └── 事件绑定 (按钮监听器)
```

## 边界情况处理

| 场景 | 处理 |
|------|------|
| 宝石池耗尽 | 无法拿取，按钮禁用 |
| 牌堆耗尽 | 市场位空缺，显示实际剩余卡数 |
| 黄金池空 | 预留卡牌时不获得黄金（仍可预留） |
| 宝石超上限 | 拿取前校验，超限阻止 |
| 预留超上限 | 最多 3 张，按钮在满时禁用 |
| 无可购买卡 | 购买按钮禁用 |
| 无可预留卡 | 预留按钮禁用 |
| 单色费用全被折扣覆盖 | 显示 ~~原价~~ ✓ |

## 已知限制 (Demo 范畴)

1. 预留卡牌在回合切换时通过遮罩保护，但遮罩透明度 85%，强光下可能可见
2. 卡牌生成为伪随机，非原版精确牌表
3. 无音效/动画
4. 无 AI 对手
5. 购买操作点击即执行，无二次确认
6. 无撤销功能
